351 PATTERNS
352 - **Optional change of behavior:** by default, the `LazyObject`
353 class forwards only one notification after recalculating and
354 silently ignores the others. In some edge cases, this could
355 lead to objects not being updated. It's now possible to enable
356 a different behavior where all notifications are forwarded; the
357 new behavior can be chosen at compile time via the configure
358 option `--disable-faster-lazy-objects` (or disabling
359 `QL_FASTER_LAZY_OBJECTS` in cmake or `userconfig.hpp`) or at run
360 time by calling
361 `LazyObject::Defaults::instance().alwaysForwardNotifications()`.
362 This might cause a slow down, so you're invited to try it out
363 and report on the mailing list. If there are no problems, the
364 new behavior might become the default in future releases. Also,
365 a new configure option `--enable-throwing-in-cycles`
366 (`QL_THROW_IN_CYCLES` in cmake or `userconfig.hpp`) is
367 optionally available; when both this option and the new behavior
368 are enabled, notifications cycles involving a lazy object will
369 throw an exception. It is suggested to try enabling the option
370 and removing such loops, if any. Thanks to Peter Caspers for
371 the change and to Ralf Konrad, Jonathan Sweemer and GitHub user
